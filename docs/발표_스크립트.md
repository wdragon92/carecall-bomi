**[슬라이드: 표지 → 문제 정의]**

> "안녕하세요, 팀 ___의 ___입니다. 저희 주제는 **독거 어르신 곁에 늘 함께 있는 상시 돌봄 AI, '보미'** 입니다.
>
> 한국 복지의 특징은 '신청주의'입니다. **알아야 받습니다.** 그런데 정작 지원이 가장 필요한 독거 어르신은 어떤 제도가 있는지 모르고, 알아도 신청 방법을 모르고, 물어볼 사람도 없습니다. 정보 비대칭에 고립이 겹친 문제입니다.
>
> 저희는 이 문제를 '전화 한 통 걸어주는 안부 서비스'가 아니라, **집집마다 AI 스피커를 놓듯 어르신 곁에 상주하는 돌봄 친구**로 풀었습니다. 보미는 매일 말동무가 되어 안부를 살피고, 대화 속에서 **건강·정서·사기·복지 신호를 실시간으로 관찰**하고, 필요한 순간 **공식 복지 자료를 검색해 근거 있는 안내**를 하고, 위급하면 **사람(119·109·보호자)에게 연결**합니다.
>
> 설계 원칙은 하나입니다 — **지어내지 않는 AI.** 금액과 자격은 LLM이 아니라 코드가 공식 자료에서 그대로 옮기고, 자료 밖 질문에는 모른다고 말합니다. 그리고 모든 AI 처리는 국내 리전 NCP CLOVA, 검색 인덱스는 서버 안 FAISS — **데이터가 밖으로 나가지 않는 소버린 AI**입니다."

## §3. 라이브 시연 — 킬러 3+1 (구현도 35점 + AI 활용도 25점) · 약 4분

**[화면 전환: https://101.79.26.62.sslip.io — 헤더 '실시간 연동' 확인, 🔊 ON, 우측 패널 보이게 창 넓게]**

### 3-1. 선인사 (10초)
접속하면 보미가 시간대에 맞춰 먼저 인사 + 음성 재생.
> "전화를 거는 게 아니라, 늘 곁에 있다가 먼저 말을 겁니다."

### 3-2. 킬러 ① 의미로 찾는다 — RAG 하이브리드 (1분 20초)
**입력(가능하면 🎤 마이크로): `무릎이 아파서 장 보러 가기가 힘들어요`**
> "'복지'라는 단어가 한 글자도 없습니다. 그런데 보미는 공감부터 하고, 무릎인공관절 수술 지원 같은 제도를 대화로 풀어 설명하고, 이어서 **📌 정보 카드**를 붙입니다. 대상·지원 내용·신청처·**정보 기준일**·**복지로 링크**까지.
>
> 포인트 두 가지입니다. 첫째, 키워드가 아니라 **의미로 찾습니다** — FAISS 벡터와 BM25를 RRF로 융합한 하이브리드 검색입니다. 둘째, **카드의 금액·신청처는 LLM이 쓴 게 아닙니다.** 코드가 공식 자료 필드에서 그대로 옮깁니다. 수치 환각이 원천 차단되는 구조고, 카드 하단 기준일이 그 증거입니다."

### 3-3. 킬러 ② 판정하고 신청까지 — 룰엔진 (50초)
**입력: `기초연금 나도 받을 수 있나?`** → 나이·가구를 여쭙는 흐름 → 가능성 판정 + 신청 준비물 패키지.
> "판정은 LLM이 아니라 **결정론적 룰**입니다. 2026년 고시값 — 단독가구 247만 원, 부부 395만 2천 원 — 은 사람이 원문을 확인하고 출처 주석과 함께 코드에 넣었습니다. 그리고 '확정'은 구청 조사의 몫이라 보미는 늘 **'가능성'** 으로만 말하고 주민센터로 안내합니다."

### 3-4. 킬러 ③ 지어내지 않는다 — 거부 게이트 (40초)
**입력: `로또 당첨 번호 좀 알려줘`** (예비: `전 국민한테 100만원씩 준다던데?`)
> "자료 밖 질문입니다. 보미는 지어내지 않고 정중히 거절합니다. 의미 점수와 어휘 증거의 **2단 게이트**인데, 실 코퍼스 평가셋 실측으로 **Hit@4 100%, 인도메인 오거부 0건, 문서 밖 거부 100%** 를 확인했습니다. 소문성 질문에 무관한 카드가 붙지 않는 것까지 별도로 방어합니다."

### 3-5. (+1) 사진 → OCR + 사기 판별 (40초)
**📎로 준비한 스미싱 문자 캡처 첨부** → 쉬운 말 설명 + "링크 누르지 마세요" + 빨간 사기 카드.
> "서류나 의심 문자를 찍어 보내면 CLOVA OCR로 읽고, 문서 종류를 분류해 쉬운 말로 설명합니다. **금액·날짜는 OCR 원문 그대로** 코드가 카드에 넣고, 스미싱이면 즉시 경고합니다. 사기는 단계를 나눠서, 이미 송금하신 정황이면 112 신고·지급정지·1332 **행동 카드**가 LLM과 무관하게 결정적으로 나갑니다."

### 3-6. 마무리 — 돌봄 리포트 (20초)
**[이야기 마치기 클릭 → 리포트 모달]**
> "대화가 끝나면 특이사항·후속 권고·안내한 복지가 정리된 **보호자용 리포트**가 나옵니다. 관찰에서 기록, 연계까지 돌봄의 한 사이클이 닫힙니다."

*(선택 — 시간 여유 시) 긴급 가드레일: `사는 게 다 부질없고 그만 살고 싶어요` → **주황(amber) 경고 배너** + 109 연결(완곡한 절망 신호라 warning — 빨강 배너는 "죽고 싶다" 직접 표현·급성 자살/몸 응급 119 전용). "AI가 해결하려 들지 않고 사람에게 연결합니다."*

## §7. 예상 Q&A (발표 준비도 20점 — 답변 준비)

| 예상 질문 | 답변 요지 |
|---|---|
| 생성 파라미터는 어떻게 설정했나 | topP 0.8·반복 페널티 1.1 고정, temperature만 역할별 차등(잡담 0.6 / RAG 접지 0.45 / 위기 0.3 / 추출 0.1). topK 샘플링은 의도적으로 미사용 — 확률 컷은 topP로 일원화 |
| 검색 top-k는? | 후보 풀 20 → RRF(k=60) 융합 → 최종 4건. 단 항목별 벡터 하한 0.45 미달이면 4건을 채우지 않고 버림 |
| 환각은 어떻게 막나 | 3계층: T1 제도 설명은 접지 / **T2 수치는 코드가 카드 삽입(LLM 발화 금지)** / T3 개인 확정은 129·주민센터 연계. + 거부 게이트, 슬롯도 발화 내 근거 토큰 없으면 LLM 값 기각 |
| 게이트 임계값 근거는? | 실측 평가셋(구어체 12문+문서밖 6문)으로 분포 확인 → 0.55 OR (0.47 AND bm25≥11). 코퍼스 재빌드 시 eval 스크립트로 재튜닝하는 절차까지 문서화 |
| LLM이 죽으면? | 기능별 mock 자동 폴백으로 대화 지속 + 안전망(119/109/사기 카드)은 애초에 LLM 비의존 규칙 기반이라 영향 없음 |
| 운영 비용은? | 인덱스 로컬(검색 ms·무료), 재임베딩은 변경분만(≈0원), LLM 호출만 종량 — 어르신 1인 일 대화량 기준 소액 |
| 개인정보는? | 전 처리 국내 리전 NCP, 인덱스 서버 내 FAISS, 이미지는 OCR 후 즉시 폐기(디스크 저장 없음) |
| 왜 모델을 둘로 나눴나 | 응답 속도(005)와 분석 깊이(007 reasoning)의 트레이드오프 — 대화는 빠르게, 추출은 비동기로 깊게 |

### Hit@1 미스 2건 상세 (실 인덱스 재현, 2026-07-07 — "17%는 틀리냐" 질문 대비)

1. **"나라에서 나오는 노인 용돈 같은 거, 나는 언제부터 받을 수 있나"** (정답 기초연금 → **3위**)
   - 1위 특별현금급여(가족요양비) vec 0.620 · 2위 노인 일자리·사회활동 지원 vec 0.620 · 3위 기초연금 vec **0.642**
   - 벡터 단독으론 기초연금이 최고점이었으나, 구어체 어휘("나라에서 주는 돈·용돈")가 다른 현금성 카드와
     BM25에서 강하게 겹쳐 RRF 융합 순위가 뒤집힘. 실대화에선 후속 턴에 "기초연금" 단어가 한 번이라도
     나오면 질의 보강으로 1위 복귀.
2. **"겨울에 난방비가 무서워서 보일러를 못 틀어"** (정답 에너지바우처 → **2위**)
   - 1위 저소득층 에너지효율개선 vec 0.479 · 2위 에너지바우처 vec 0.468 · 3위 연탄쿠폰 vec 0.429
   - RRF 융합 점수는 1·2위 동점(0.0325), 벡터 차 0.011 — 1~3위 전부 저소득 난방 지원 계열,
     "정답의 형제 서비스"가 근소하게 앞선 것.

**공통 방어 논리**: 미스 방향이 무관 자료가 아니라 같은 도메인의 이웃 서비스이고, 보미는 상위 4건을
카드로 접지(Hit@4 100%)하므로 실제 안내 단위에서는 12/12 전부 성공.

### temperature 근거 방어 (1차 발표 강사 지적 대응)

**지적**: topP 0.8·RP 1.1은 공식 권장 범위로 설명되지만, temperature 4단(0.6/0.45/0.3/0.1)의
타당성 근거가 부족 — boundary 커버리지 같은 기준으로 검증했는가?

**방어 3단** (질문의 층위를 나누는 게 핵심):

1. **층위 구분** — 온도 경계값을 전수 탐색해 모델 출력 품질을 재는 것은 파운데이션 모델
   벤치마킹의 영역. 애플리케이션이 책임질 질문은 "**서비스의 정확성 계약이 temperature에
   민감한가**"이고, 저희는 민감하지 않도록 설계했다.
2. **구조적 근거** — 이 시스템에서 temperature는 정확성 파라미터가 아니라 **문체 파라미터**다:
   금액·수치는 LLM 발화 금지+코드 카드 삽입+접지 턴 금액 하드가드(정규식 치환), 연락처는
   상수+발화 백스톱, 자격 판정은 결정론 룰엔진, 근거 카드는 서비스명 매칭 게이트, 추출 실패는
   규칙 안전망이 보존. **온도를 잘못 골라도 '틀린 사실'이 어르신에게 도달하는 경로가 코드로
   차단**되어 있다. 그 위에서 값은 공식 기본값 0.5를 앵커로 역할 방향에 따라 소폭 조정:
   잡담 0.6(화제 다양성) / 접지 0.45(카드-답변 일치) / 위기 0.3(지침 이탈 억제) / 추출 0.1(JSON 재현성).
3. **실증(보강 가능)** — "커버리지"를 서비스 지표로 재정의해 sweep 측정: 접지 턴에서
   temp {0.2, 0.45, 0.7, 0.9} × N회 반복 → ① 카드-답변 서비스명 일치율 ② 금액 하드가드
   발동률(수치 누출 시도 빈도) ③ 지침 준수(문장 수) / 추출에서 temp {0.1, 0.5, 0.9} →
   ④ JSON 파싱 성공률 ⑤ 동일 입력 반복 시 필드 재현성. — 모델 벤치마킹이 아닌
   **서비스 계약 기준의 온도 민감도 측정**이라 지적에 정면 대응이 된다.

**한 줄 답변**: "temperature가 틀려도 사실이 틀리지 않게 만든 구조가 저희의 1차 근거이고,
값 자체는 공식 기본값을 앵커로 역할별 방향 조정입니다. 지적 주신 커버리지는 서비스 지표
